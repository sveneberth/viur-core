20 adminInfo:Union[Dict[str, Any], Callable] = None 

21 """ 

22 A ``dict`` holding the information necessary for the Vi/Admin to handle this module. If set to 

23 ``None``, this module will be ignored by the frontend. The currently supported values are: 

24 

25 name: ``str`` 

26 Human-readable module name that will be shown in Vi/Admin 

27 

28 handler: ``str`` (``list``, ``tree`` or ``singleton``): 

29 Which (proto-)type is used, to the frontend can 

30 initialize it handler correctly. 

31 

32 icon: ``str`` 

33 (Optional) The name (eg "icon-add") or a path relative the the project 

34 (eg. /static/icons/viur.svg) for the icon used in the UI for that module. 

35 

36 columns: ``List[str]`` 

37 (Optional) List of columns (bone names) that are displayed by default. Used only 

38 for the list handler. 

39 

40 filter: ``Dict[str, str]`` 

41 (Optional) Dictionary of additional parameters that will be send along when 

42 fetching entities from the server. Can be used to filter the entities being displayed on the client-side. 

43 

44 display: ``str`` ("default", "hidden" or "group") 

45 (Optional) "hidden" will hide the module in the main bar 

46 (itwill not be accessible directly, however it's registered with the frontend so it can be used in a 

47 relational bone). "group" will show this module in the main bar, but it will not be clickable. 

48 Clicking it will just try to expand it (assuming there are additional views defined). 

49 

50 preview: ``Union[str, Dict[str, str]]`` 

51 (Optional) A url that will be opened in a new tab and is expected to display 

52 the entity selected in the table. Can be “/{{module}}/view/{{key}}", with {{module}} and {{key}} getting 

53 replaced as needed. If more than one preview-url is needed, supply a dictionary where the key is 

54 the URL and the value the description shown to the user. 

55 

56 views: ``List[Dict[str, Any]]`` 

57 (Optional) List of nested adminInfo like dictionaries. Used to define 

58 additional views on the module. Useful f.e. for an order module, where you want separate list of 

59 "payed orders", "unpayed orders", "orders waiting for shipment", etc. If such views are defined, 

60 the top-level entry in the menu bar will expand if clicked, revealing these additional filters. 

61 

62 actions: ``List[str]`` 

63 (Optional) List of actions supported by this modules. Actions can be defined by 

64 the frontend (like "add", "edit", "delete" or "preview"); it can be an action defined by a plugin 

65 loaded by the frontend; or it can be a so called "server side action" (see "customActions" below) 

66 

67 customActions: ``Dict[str, dict]`` 

68 (Optional) A mapping of names of server-defined actions that can be used 

69 in the ``actions`` list above to their definition dictionary. See .... for more details. 

70 

71 disabledActions: ``List[str, dict]`` 

72 (Optional) A list of disabled actions. The frontend will inject default actions like add or edit 

73 even if they're not listed in actions. Listing them here will prevent that. It's up to the frontend 

74 to decide if that action won't be visible at all or it's button just being disabled. 

75 

76 sortIndex: ``int`` 

77 (Optional) Defines the order in which the modules will appear in the main bar in 

78 ascrending order. 

79 

80 indexedBones: ``List[str]`` 

81 (Optional) List of bones, for which an (composite?) index exists in this 

82 view. This allows the fronted to signal the user that a given list can be sorted or filtered by this 

83 bone. If no additional filters are enforced by the :meth:`listFilter<viur.core.prototypes.list.listFilter>` 

84 and ``filter`` is not set, this should be all bones which are marked as indexed. 

85 

86 changeInvalidates: ``List[str]`` 

87 (Optional) A list of module-names which depend on the entities handled 

88 from this module. This allows the frontend to invalidate any caches in these depended modules if the 

89 data in this module changes. Example: This module may be a list-module handling the file_rootNode 

90 entities for the file module, so a edit/add/deletion action on this module should be reflected in the 

91 rootNode-selector in the file-module itself. In this case, this property should be set to ``["file"]``. 

92 

93 moduleGroup: ``str`` 

94 (Optional) If set, should be a key of a moduleGroup defined in .... . 

95 

96 editViews: ``Dict[str, Any]`` 

97 (Optional) If set, will embed another list-widget in the edit forms for 

98 a given entity. See .... for more details. 

99 

100 If this is a function, it must take no parameters and return the dictionary as shown above. This 

101 can be used to customize the appearance of the Vi/Admin to individual users. 

102 """ 

103

106 """ 

107 If set, a list of access rights (like add, edit, delete) that this module may support. 

108 These will be prefixed on instance startup with the actual module name (becomming file-add, file-edit etc)

113 def __init__(self, moduleName, modulePath, *args, **kwargs): 

114 self.moduleName = moduleName #: Name of this module (usually it's class name, eg "file") 

115 self.modulePath = modulePath #: Path to this module in our URL-Routing (eg. json/file") 

116 self.render = None #: will be set to the appropriate render instance at runtime 

117 

118 if self.adminInfo and self.accessRights: 

119 for r in self.accessRights: 

120 rightName = "%s-%s" % (moduleName, r) 

121 

122 if not rightName in conf["viur.accessRights"]: 

123 conf["viur.accessRights"].append(rightName)

125 def _resolveSkelCls(self, *args, **kwargs) -> Skeleton: 

126 """ 

127 Retrieve the generally associated :class:`viur.core.skeleton.Skeleton` that is used by 

128 the application. 

129 

130 This is either be defined by the member variable *kindName* or by a Skeleton named like the 

131 application class in lower-case order. 

132 

133 If this behavior is not wanted, it can be definitely overridden by defining module-specific 

134 :func:`~viewSkel`, :func:`~addSkel`, or :func:`~editSkel` functions, or by overriding this 

135 function in general. 

136 

137 :return: Returns a Skeleton class that matches the application. 

138 """ 

139 

140 return skeletonByKind(self.kindName if self.kindName else str(type(self).__name__).lower())

148

150 """ 

151 return self._resolveSkelCls(*args, **kwargs)()
